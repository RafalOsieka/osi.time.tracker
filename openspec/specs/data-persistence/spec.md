## Requirements
### Requirement: REQ-NFR-010 Drizzle as the data-access layer
The project SHALL use Drizzle (`drizzle-orm` with the `postgres` driver) as the sole data-access layer for the PostgreSQL database, and `drizzle-kit` as the development tool for generating migrations. Application server code SHALL access the database only through the shared Drizzle client; raw driver usage outside that client SHALL NOT be introduced.

**Status**: Not met
**Evidence** — none

#### Scenario: Server context obtains a Drizzle client
- **WHEN** server-side code (Nitro context) needs database access
- **THEN** it imports a single shared Drizzle client module that is initialized from the `DATABASE_URL` environment variable

#### Scenario: Missing database configuration
- **WHEN** the application starts and `DATABASE_URL` is not set
- **THEN** initialization SHALL fail fast with a clear error rather than connecting to a default or silently continuing

### Requirement: REQ-NFR-011 Committed SQL migration workflow
Schema changes SHALL be expressed as SQL migration files generated by `drizzle-kit generate` and committed to the repository. Migrations SHALL be applied via the `drizzle-orm` migrator. The project SHALL expose `db:generate` and `db:migrate` scripts in `package.json`.

**Status**: Not met
**Evidence** — none

#### Scenario: Generating a migration
- **WHEN** a developer runs the `db:generate` script after a schema change
- **THEN** `drizzle-kit` produces a new SQL migration file under the migrations directory that is committed to version control

#### Scenario: Applying migrations
- **WHEN** the `db:migrate` script runs against a reachable database
- **THEN** the `drizzle-orm` migrator applies all pending migrations in order and records them so they are not re-applied

#### Scenario: Migration run with no pending migrations
- **WHEN** `db:migrate` runs and all migrations are already applied
- **THEN** the migrator SHALL complete successfully without making changes

### Requirement: REQ-NFR-012 Unattended migrations under Docker Compose
Database migrations SHALL run unattended during `docker compose up`, executing before the application begins serving traffic. The migration step SHALL be a dedicated step (separate from the request-handling process) and SHALL halt startup on migration failure.

**Status**: Not met
**Evidence** — none

#### Scenario: Successful startup applies migrations first
- **WHEN** the stack is started with `docker compose up` against a fresh database
- **THEN** pending migrations are applied to completion before the app accepts requests

#### Scenario: Migration failure blocks the app
- **WHEN** a migration fails during startup
- **THEN** the startup sequence SHALL stop with a non-zero exit and the application SHALL NOT begin serving traffic

### Requirement: REQ-NFR-013 Migration tooling verification
The migration workflow SHALL be verified end-to-end using a throwaway fake table before any domain schema is introduced. The verification artifact SHALL be removed before domain schema work begins and SHALL NOT remain in the committed schema.

**Status**: Not met
**Evidence** — none

#### Scenario: End-to-end generate then migrate
- **WHEN** a temporary fake-table schema is defined and `db:generate` then `db:migrate` are run against a database
- **THEN** the fake table exists in the database, proving the generate-to-migrate pipeline works

#### Scenario: Verification artifact removed
- **WHEN** verification is complete
- **THEN** the fake table definition and its verification-only migration SHALL be removed so the committed schema contains no throwaway tables
